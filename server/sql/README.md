# SQL Schema and Query Files

This directory contains the raw SQL used by the server during the course SQL-first phase.

The goal is to keep the database definition and demo query logic visible as `.sql` files while the Elysia backend loads and executes those files without using Drizzle ORM queries.

## How It Works

- `schema.sql` defines the database structure (tables, constraints, etc.)
- `seed.sql` contains initial data for populating the database
- `queries/` contains route-level SQL statements used by backend services
- The migration utility (`src/db/migrate.ts`) executes `schema.sql` directly through the `postgres` package
- The seeding utility (`src/db/seed-sql.ts`) executes `seed.sql` directly through the `postgres` package
- Backend route services execute query files with `sql.file(...)`
- Drizzle is only used to generate TypeScript table metadata/types from the live SQL-created database

## Directory Structure

```txt
sql/
  schema.sql
  seed.sql
  queries/
    categories/
      get-all.sql
      get-by-id.sql

```

## Database Workflow

### To run migrations:

```bash
bun run reset   # Drops the database
bun run migrate # Executes schema.sql to create tables
```

### To seed the database:

```bash
bun run seed    # Executes seed.sql to populate with test data
```

### To modify the database schema:
1. Edit `schema.sql` in this directory
2. Run `bun run reset` to clear the database
3. Run `bun run migrate` to apply schema changes
4. Run `bun run seed` to populate with test data (optional)

### To modify seed data:
1. Edit `seed.sql` in this directory
2. Run `bun run seed` to populate with updated test data

## Query File Workflow

Route query files live under `queries/<feature>/`.

Example:

```sql
-- queries/categories/get-by-id.sql
SELECT id, name, image, status
FROM category
WHERE id = $1 AND status = 'Active'
LIMIT 1;
```

The backend loads this file from the service layer:

```typescript
const [category] = await sql.file<CategoryRow[]>(
  sqlFile("categories", "get-by-id.sql"),
  [id],
);
```

Use positional parameters (`$1`, `$2`, `$3`, etc.) inside SQL files. Pass the parameter values as the second argument to `sql.file(...)`.

Do not concatenate user input into SQL strings. Keep all user-provided values as parameters.

## TypeScript Types

Raw SQL files are the source used for schema and queries, but TypeScript cannot infer types directly from `.sql` files.

To get TypeScript row types:

1. Apply `schema.sql` to the database.
2. Run Drizzle introspection against the live database.
3. Use the generated Drizzle schema only for TypeScript types.

Current type bridge:

```txt
src/db/generated/schema.ts  # generated by drizzle-kit introspect
src/db/types.ts             # readable exported row/insert types
```

Route services should import those types with `import type`, while still executing raw SQL files.

## Notes

- The `migrate` command only executes `schema.sql`
- The `seed` command only executes `seed.sql`
- Route SQL files in `queries/` are executed by backend services, not by the migration command
- These are separate operations so you can apply schema changes without re-seeding
- The `DATABASE_URL` environment variable must be set in `.env`
- Tables use `SERIAL PRIMARY KEY` for auto-incrementing IDs
- Foreign key constraints are enforced, so ensure referenced data exists
- PostgreSQL folds unquoted table names to lowercase, so `CREATE TABLE Category` is queried as `category`
